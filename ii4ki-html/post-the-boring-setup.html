The boring setup that prevents disaster

2026-05-07 7 min read 1,420 words

I have a rule for personal projects: if the setup is interesting, I've already lost. The setup should be so boring that I can describe it on a sticky note, and so recoverable that I can restore it on a fresh laptop in under an hour. Anything sexier than that is a future incident I haven't had yet.

This post is about what that looks like in practice. Not a guide. Just a description of what I currently do, what I used to do, and why I changed.

The four-layer secrets thing

Every secret in my life lives in exactly one of four places, in this order of preference:

  1. Environment variable, scoped to a single shell session. Throwaway tokens, temporary API keys, anything I'm going to rotate the same day.
  2. OS keychain, via a thin secrets shell wrapper. Long-lived but boring: GitHub tokens, npm tokens, AWS keys for personal accounts.
  3. Age-encrypted file in a private repo. Things I need on more than one machine but don't want in a cloud password manager.
  4. Hardware token. The keys that, if leaked, would ruin my month. Domain registrar 2FA, root keys for the encrypted file in layer 3, recovery codes.

The interesting thing about this list is that layer 1 is where almost everything starts, and almost nothing stays. A new API key gets pasted into a shell for a few minutes while I'm trying something. If it survives the experiment, it graduates to layer 2. If it ends up on a second machine, layer 3. If losing it would be catastrophic, layer 4.

The graduation step is manual on purpose. I don't want a script that automatically promotes secrets. I want to think about it every time.

What the wrapper actually does

The shell wrapper is twelve lines of bash. It is the only piece of "infrastructure" in the entire system:

# ~/.local/bin/secrets
case "$1" in
  get) security find-generic-password -s "$2" -w ;;
  set) security add-generic-password -U -s "$2" -a "$USER" -w "$3" ;;
  ls)  security dump-keychain | grep '^    "svce"' | sort -u ;;
  *)   echo "usage: secrets {get|set|ls} <name> [value]" ;;
esac

That's it. No daemon. No node_modules. No "first, install our CLI". When the laptop dies, I install macOS, restore the keychain from a Time Machine backup, and the wrapper works again. The wrapper itself is in my dotfiles, which are public, because there is nothing in it worth hiding.

The best part of a boring system is that it doesn't get a chapter in your post-mortem.

Recoverable defaults

The other half of "boring" is "recoverable". For every machine I use, I keep a single bootstrap script in a public dotfiles repo. It does three things:

  • Installs a small fixed list of packages (homebrew, the usual suspects).
  • Symlinks every dotfile from the repo into ~/.
  • Prints a numbered list of things I have to do by hand: log into the password manager, plug in the hardware token, restore the keychain.

The list of manual steps is in the script, as comments, in the order I have to do them. It's not magic. It's a checklist with a fancy hat.

What I changed, and why

For about two years I ran a much fancier version of this. Encrypted vaults synced through a self-hosted thing, a custom CLI in Go, hooks that auto-promoted secrets between layers, the works. It was fun to build. It was a nightmare to recover.

When I had to restore it on a new machine, I spent a full evening fixing it. Half the issues were things the fancy CLI was doing for "convenience": writing config to a path that no longer existed, assuming a daemon was already running, requiring a network call to a service I hadn't restored yet.

The current setup took me one afternoon to write. I've restored it twice. The second time took twenty minutes. That's the entire pitch.

The takeaway, if you want one

I'm not saying don't build interesting things. I'm saying don't build interesting things in the layer that has to keep working when everything else is on fire. Put your creativity in the layer above. Make the load-bearing parts boring enough that you can describe them on a sticky note.

The full bootstrap script and the age-encrypted layer are in the secrets-system repo. There is, deliberately, nothing clever in it.
